開発ロードマップ¶

このドキュメントは Fabric の開発予定の概要です。` <http://docs.fabfile.org/en/latest/roadmap.html> の最新のドキュメント`を読んでください!

警告

ここの情報は予告なしに変更されます。従って人生やキャリアに関わるどんな決定のベースにもすべきではありません!

Fabric 1.x¶

Fabric 1.x はまだ現役ですが内部の技術的負の遺産や下位互換性を犠牲にすることなしに大幅な改良ができないため,限界点に達しています。

そのようなわけで将来の 1.x リリース(1.6 以前)は小中規模の改造(大規模なオーバーホールを必要としない新機能)とバグフィックスに重点が置かれます。

Invoke, Fabric 2.x, Patchwork¶

1.xは上記の方針で進めますが、我々はコンセプトを再構築した 2.x バージョンのツールに取り組みます。計画は:

  • Invoke ツール/ライブラリ<https://github.com/pyinvoke/invoke>`の完成とリリース(参照 :issue:`565)、これは改良したFabricのタスク実行コンポーネントのスタンドアロン・バージョン。

    • 2013年後半現在、Invoke は成熟期に近づいてすでに Fabric に欠けている機能を少し持っています。それだけでなく:

      • さらにわかりやすく強力な名前空間の実装

      • “regular” スタイル CLI フラッグ、

      • before/after フック

      • わかりやすいコンテキスト管理

    • Invoke の新しい依存性の少ないコードベースによりすでに Python 3 と互換性があります。

    • Fabric 2 が開発されたので Invoke は高品質のスタンドアロン・タスクランナーとして Fabric の要求に合わせて成長を続けます。

  • Fabric 2.0 の構成、ほとんど書きなおされた Fabric コア:

    • タスクの実行にもっと Invoke を活用する。Fabric自身をもっとライブラリ志向に。

    • オブジェクト指向の hosts/host リストの実装とそれに関すること(例えば不完全な host 文字列の廃止と直感的でない env 変数の取り扱い。)

    • デフォルトで共有状態を使わない。(Invokeのコンテキストデザインに感謝)

    • 下位互換性を難しくしている他のコアのオーバーホール。

    • テスト駆動開発(Invokeと同様)

    • Paramiko に Python 3 と互換性を持たせる必要性 - 最もやっかいな問題の一つ。 参照 Paramiko #16.

  • fabric.contrib.* のスタンドアロン “super-Fabric” へのスピンオフ。(上記の)ライブラリ、Patchwork.

    • これで “Fabricコアを使う一般的に有効なショートカット”と別にコアが”ホスト上のコマンド実行”を繰り返すことができます。

    • #461 の多くの作業準備と先行技術の調査は完了

    • APIを考えた Patchwork のアルファ版の公開コードはありますが、これは Fabric 1.xがベースです。Fabric 2.x が安定するまでに Fabric 2.x ベースになるでしょう。
